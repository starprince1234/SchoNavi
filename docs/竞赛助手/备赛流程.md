# 高校竞赛通用备赛流程

> 本流程适用于学术论文、工程制作、程序设计、创新创业和案例分析类赛事。具体材料、匿名、AI 和提交规则必须服从当届章程。

## 一、启动：先判断值不值得参加

### 1. 获取四份文件

1. 当届官方通知或邀请函。
2. 竞赛章程/参赛规则。
3. 评审规则、赛题说明或命题清单。
4. 本校竞赛认定、经费报销和请假政策。

没有这四类信息时，不要先交费或购买培训。聚合平台的报名页不能替代主办方规则。

### 2. 建立赛事卡片

|字段|要记录的内容|
|---|---|
|赛事身份|正式全名、届次、主办单位、官网|
|资格|学历、年级、专业、年龄、学籍、地区|
|组队|人数、是否同校、指导教师、成员变更截止|
|赛制|校赛/省赛/国赛，线上/线下，初赛/复赛/答辩|
|材料|论文、代码、数据、视频、PPT、实物、证明|
|时间|校内截止、官方截止、时区、比赛和答辩日期|
|费用|报名、材料、差旅、设备，退费和报销|
|合规|匿名、查重、AI、知识产权、数据和伦理|
|校内认定|赛事等级、加分对象、是否只认最高奖|

### 3. Go/No-Go 决策

以下任一项为“否”，应暂停报名：

- 队伍符合资格且能覆盖完整赛程。
- 有足够时间做真实成果，而不是只包装材料。
- 关键设备、数据、场地或企业授权可获得。
- 团队接受知识产权、成果公开和奖金分配规则。
- 赛事费用和收益预期合理，本校认定已核实。

## 二、组队：按交付物配置能力

|赛事类型|核心角色|最低能力组合|
|---|---|---|
|数学/统计建模|建模、计算、写作|模型选择与验证、可复现代码、论文表达|
|程序设计|算法、实现、调试|数据结构与算法、复杂度、比赛环境熟练度|
|工程制作|硬件、软件/控制、测试|原理设计、加工装配、调试和测量|
|学术调查|研究设计、数据、写作|抽样/实验、统计分析、学术规范|
|创新创业|产品/技术、市场/运营、财务/表达|可运行成果、真实验证、证据化路演|
|设计艺术|研究、创意、制作、呈现|原创性、工艺/媒介、视觉叙事|

### 团队协议至少写清

- 每人的任务、预计投入和不可用时间。
- 代码、数据、图纸、论文和账号的共享方式。
- 署名顺序、成果归属、专利申请、奖金和费用。
- 退出、替补、延期和冲突处理。
- 禁止代做、挂名、伪造数据和违规使用 AI。

## 三、立项：把题目变成可验证目标

### 1. 问题定义

用一页纸回答：谁遇到什么问题、现状如何、为什么重要、已有方案为何不足、团队要改善哪个可测指标。

### 2. 基线与验收

没有基线就无法证明创新。按项目类型至少设置：

- 算法：准确率、召回率、延迟、内存、鲁棒性和公开基线。
- 硬件：精度、功耗、稳定性、成本、环境范围和安全。
- 社科：样本覆盖、量表信效度、识别策略和稳健性。
- 创业：真实用户、留存/复购、合同回款、单位经济性。
- 设计：用户任务、可用性、生产工艺、无障碍和可持续性。

### 3. 风险登记

|风险|早期信号|预案|
|---|---|---|
|数据拿不到|授权长期未确认|准备公开数据或缩小问题|
|硬件延期|核心器件未到货|准备替代器件和仿真方案|
|模型无提升|只在训练集有效|回到基线、数据和验证设计|
|成员掉线|连续两周无交付|拆小任务、提前替补或缩范围|
|规则变化|官网发布新通知|指定一人每周巡检并记录变更|
|提交失败|文件过大/平台不熟|提前做完整提交演练|

## 四、执行：用证据驱动迭代

### 每周循环

1. 明确本周一个可验证目标。
2. 完成实验/开发并保留原始记录。
3. 与基线比较，不只展示最好结果。
4. 写下失败原因和下一步决策。
5. 同步更新论文、图表和证据索引。

### 推荐目录

```text
project/
├─ rules/          # 当届通知、章程、变更记录
├─ data/           # 数据说明、许可、原始/处理后数据索引
├─ src/            # 代码、脚本、配置和依赖
├─ experiments/    # 实验日志、参数、随机种子、结果
├─ hardware/       # 图纸、BOM、固件、测试记录（如适用）
├─ evidence/       # 专利、检测、合同、用户验证等证明
├─ paper/          # 论文、参考文献、图表
├─ pitch/          # PPT、视频脚本、答辩问答
└─ submission/     # 最终提交副本和回执
```

个人敏感信息、密钥、真实合同和未脱敏数据不得提交到公开仓库。

## 五、阶段评审

### 第一次评审：方向是否成立

- 问题真实且范围可控。
- 数据/器件/用户可获得。
- 验收指标和基线明确。
- 与往届作品相比有可解释差异。

### 第二次评审：成果是否可信

- 核心流程已经跑通，不依赖赛前最后一周“集成”。
- 结果可复现，有失败案例、误差和边界。
- 学生能解释每个关键决策。
- 引用、数据授权、伦理和知识产权无明显缺口。

### 第三次评审：提交是否合规

- 逐条对照规则，而不是凭往届模板。
- 匿名、页数、命名、大小、格式、时区都正确。
- 论文、代码、视频、PPT和实物数据一致。
- AI 使用声明完整，引用可访问且确实支持正文。

## 六、冲刺与提交

### T-14 天

- 冻结大方向，只修复高价值问题。
- 做一次完整答辩或限时模拟。
- 邀请非项目成员按评审标准盲审。

### T-7 天

- 冻结核心数据和模型版本。
- 全量检查图表、单位、编号、引用和证明材料。
- 在干净设备上复现代码/部署。

### T-48 小时

- 生成最终候选包，完成上传演练。
- 检查文件哈希、页数、大小和匿名信息。
- 准备主网络、备用网络和双设备。

### T-6 小时

- 只修阻断提交的问题，不再重构。
- 提前提交；平台允许时下载回读。
- 保存回执、提交时间、最终文件和哈希。

## 七、答辩

推荐叙事顺序：问题与证据 → 现有方案不足 → 核心创新 → 实现 → 对照实验/用户验证 → 局限与下一步。

每个关键数字都应能回答：数据来自哪里、样本多少、如何计算、与什么比较、误差多大。答不出的市场规模、性能提升或社会效益数字应删除。

## 八、赛后复盘

无论是否获奖，在一周内完成：

- 记录规则变化、评审反馈和实际赛程。
- 区分方向错误、执行问题、表达问题和随机因素。
- 整理可公开成果与必须保密/删除的数据。
- 明确论文、专利、开源、产品化或下一赛事的去向。
- 更新本知识库中的过期时间和失效链接。
