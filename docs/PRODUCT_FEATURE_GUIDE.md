# SchoNavi 功能说明（当前 App 修订版）

> 本文面向评委、体验用户、产品和研发沟通使用。内容基于当前 Flutter App 与 Android 原生能力整理，重点说明用户能完成什么、为什么这样设计，以及小组件、系统日历、通知等能力接入后对交互和体验的改进。
> App 支持 mock / local / HTTP / AI 多种数据源形态；Android 原生能力仅在 Android 设备上生效，其他平台会安全降级。

## 一、产品定位

SchoNavi 是一款面向高校学生的 AI 学业与竞赛导航助手。它试图解决三类高频但割裂的问题：

- 找导师时，信息分散、难判断匹配度、缺少第一封联系邮件的表达模板。
- 选竞赛时，规则、报名节点、适合人群和备赛路径分散在多个站点和文档里。
- 开始备赛后，计划容易停留在 App 内，无法进入用户每天真正会看的桌面、通知和日历。

因此，SchoNavi 的产品主线不是“问一次 AI 得一个答案”，而是把“提出目标 → 推荐候选 → 解释依据 → 生成行动材料 → 制定计划 → 系统级提醒 → 持续反馈调整”做成一条闭环。

引入 Android 原生能力后，备赛体验从“打开 App 才能看到计划”扩展为“桌面能看进度、日历能记住截止、通知能推动当天任务、点击能回到计划详情”。这让 AI 生成的计划更接近真实执行工具，而不是一份静态建议。

## 二、核心设计思路

### 1. 用自然语言降低开始成本

用户不需要先理解筛选器、字段或规则库。首页用一个输入框承接“找导师”和“找竞赛”两类意图，用户像提问一样输入目标，系统再转成推荐、计划或追问。

### 2. 推荐必须可解释

导师和竞赛推荐都强调“为什么推荐”和“依据来自哪里”。这避免 AI 只输出结论，也方便用户、评委或指导教师判断推荐是否可信。

### 3. AI 给建议，关键改动由用户确认

备赛 AI 助手不会直接改计划，而是生成改动卡。用户逐张接受或拒绝后，改动才写入计划。这样既让 AI 有执行力，又保留用户对时间安排的最终控制权。

### 4. 原生能力承担“低摩擦触达”

App 内详情页适合深度管理；Android 小组件、通知和日历适合日常触达。三者分工明确：

- 小组件负责“抬眼可见”：倒计时、进度、当前阶段、下一项任务。
- 系统日历负责“重要节点不丢”：报名截止、提交截止、比赛开始。
- 通知负责“当天推动”：今日摘要、下一任务、完成/稍后提醒/查看计划。

### 5. 权限和失败路径要有回退

日历写入、通知、桌面小组件都依赖系统能力。App 在不支持或权限不足时不阻塞主流程，而是提示用户手动添加、打开系统日历确认保存，或回到 App 内计划管理。

## 三、功能全景

| 模块 | 用户能做什么 | 体验细节 |
|---|---|---|
| 首页双入口 | 用自然语言发起导师或竞赛需求 | “找导师 / 找竞赛”共用输入框，减少入口选择成本 |
| 对话式导师推荐 | 获得导师卡片、继续追问、进入详情 | 多轮追问保持上下文，围绕当前导师不跑偏 |
| 导师详情 | 查看导师基础信息、研究方向、推荐依据和数据来源 | 让推荐从“像答案”变成“可核查的候选” |
| 匹配分析 | 基于个人档案分析学生与导师的契合度 | 用维度分析辅助判断，不预测录取概率 |
| 套磁邮件 | 生成可编辑邮件主题和正文 | 结合导师事实与学生背景，定位为草稿而非自动发送 |
| 导师对比 | 横向比较 2 到 3 位导师 | 适合从候选池中确定优先联系顺序 |
| 竞赛推荐 | 根据兴趣、专业、年级、投入时间推荐竞赛 | 推荐结果连接到规则摘要和备赛计划 |
| 个性化备赛计划 | 生成分阶段任务、倒计时、智能日历 | 结合目标日期、时间投入、经验水平和赛事类型 |
| 水平诊断 | 让 AI 判断当前基础并影响计划密度 | 新手补基础，进阶用户减少重复铺垫 |
| 备赛 AI 助手 | 用自然语言调整计划 | 输出改动卡，用户确认后落地 |
| Android 桌面小组件 | 在系统桌面查看备赛倒计时和进度 | 四档尺寸自适应，多计划自动轮换，点击回计划 |
| 系统日历集成 | 把报名截止、提交截止或比赛开始加入日历 | 权限允许时直接写入，不允许时打开日历 App 由用户确认 |
| Android 通知与提醒 | 推送任务、摘要和截止提醒 | 通知动作支持完成任务、稍后提醒、查看计划 |
| 个人档案 | 填写学校、专业、GPA、排名、兴趣、经历 | 提升推荐、匹配、邮件和备赛计划的个性化程度 |
| 收藏与历史 | 管理导师候选池，回看搜索和对话 | 支持持续使用，不是一次性问答 |
| 反馈 | 对 AI 回复或推荐卡片提交反馈 | 用于记录“不准、不相关、信息错误”等问题 |

## 四、核心使用流程

### 1. 首页：一个入口承接两类目标

用户进入 App 后，可以直接选择“找导师”或“找竞赛”。两个入口共用自然语言输入框，适合低门槛开始：

- “我想做计算机视觉方向，最好是 985 高校导师。”
- “我零基础，想参加适合大二学生的人工智能竞赛。”
- “我对数学建模感兴趣，每周大概能投入 8 小时。”

首页的价值是把用户从“我该点哪个功能”带到“我先说出目标”。后续系统再根据意图进入导师推荐、竞赛推荐或计划创建链路。

### 2. 对话式导师推荐

用户输入研究兴趣、学校偏好、升学阶段或个人背景后，系统会先理解需求，再给出导师推荐卡片。卡片通常包含导师基础信息、研究方向、匹配等级和推荐理由。

用户可以继续追问：

- “为什么推荐这位老师？”
- “有没有更偏强化学习的导师？”
- “这几位老师谁更适合我现在的背景？”

当用户围绕某位导师继续追问时，对话会保持当前导师上下文。这样多轮对话不会把“当前在讨论哪位导师”弄混，也方便用户在详情、推荐卡片和对话之间来回切换。

### 3. 导师详情、匹配分析与行动材料

点击导师卡片后，用户进入导师详情页。详情页不只展示姓名、主页、研究方向和简介，还强调推荐依据与数据来源，让用户知道这位导师为什么被推荐。

在详情页或相关入口中，用户可以继续完成三类行动：

- 生成套磁邮件：输出主题和正文草稿，用户自行修改后复制发送。
- 查看匹配分析：结合个人档案与导师事实，分析研究方向、经历背景和准备程度。
- 收藏导师或加入对比：建立候选池，之后比较 2 到 3 位导师的优劣和准备建议。

这里的设计重点是把“推荐”变成“下一步可执行”。用户不只是看到名单，还能判断、比较、准备联系材料。

### 4. 竞赛推荐与详情

用户描述竞赛方向、专业基础、年级和可投入时间后，系统推荐适合的竞赛，并在详情页展示规则摘要、报名时间、比赛时间、赛制、组队要求、主办方和官方链接。

竞赛详情连接到“创建备赛计划”。这让用户从“找到一个竞赛”自然过渡到“我该怎么准备”，避免推荐结果停留在信息浏览层面。

### 5. 个性化备赛计划

用户创建计划时填写目标日期、每周投入时间、经验等级和赛事时间模型。系统会生成分阶段备赛计划，包括：

- 阶段目标与时间安排。
- 必做任务、可选任务和用户自定义任务。
- 每个阶段的准备重点和个性化建议。
- 倒计时、阶段时间线、任务清单和截止节点。

计划生成不是简单模板套用。系统会根据赛事类型区分“提交型”和“窗口型”：

- 提交型竞赛关注作品提交截止，可能保留答辩准备阶段。
- 窗口型竞赛关注比赛开始和结束时间，更强调集中冲刺。

当目标日期变化时，计划会重新排布未完成任务，同时尽量保留已完成任务、备注和用户已做出的修改，减少“改一次日期就推倒重来”的挫败感。

### 6. 水平诊断

创建计划过程中，用户可以让 AI 根据个人背景诊断当前水平。诊断结果会影响计划密度和任务选择：

- 初学者需要更多基础补齐任务。
- 有经验用户可以减少重复学习，把时间放在实战、组队、复盘和提交质量上。
- 时间很紧时，系统会给出“时间偏紧”或“任务超负荷”的提示，帮助用户尽早调整目标。

这个能力的价值在于让备赛计划更接近真实起点，而不是所有人都拿到同一份通用日程。

### 7. 备赛 AI 助手

在计划详情页，用户可以唤出备赛 AI 助手，用自然语言调整计划：

- “把组队阶段往后挪一周。”
- “下周考试比较多，帮我降低任务量。”
- “加一次模拟答辩。”
- “把算法学习拆成更小的任务。”

AI 助手会生成改动卡，覆盖移动任务、新增任务、删除任务、重排阶段、追加建议等类型。每张卡说明要改什么、为什么这样改。用户可以逐张接受或拒绝。

这是一种“人机协作”设计：AI 负责提出结构化方案，用户负责最终审批。它比纯聊天更可执行，也比自动改日程更安全。

## 五、Android 原生能力带来的体验升级

### 1. 桌面小组件：从“打开 App 查看”到“桌面抬眼可见”

备赛列表页右上角提供“添加桌面小组件”入口。支持一键向系统发起添加请求；如果设备不支持一键添加，会提示用户从系统桌面长按手动添加 SchoNavi 小组件。

当前小组件支持四档尺寸，适配用户在桌面上的不同空间：

| 尺寸 | 主要内容 | 适合场景 |
|---|---|---|
| Micro | 竞赛名、D-Day、进度细线 | 桌面空间很小，只看倒计时 |
| Small | 倒计时、当前阶段、下一项任务、完成进度 | 默认推荐尺寸，信息密度适中 |
| Wide | 左侧倒计时与阶段，右侧下一任务和进度 | 横向桌面布局，适合快速扫读 |
| Hero | 大号倒计时、进度、阶段轴、连续推进状态、下一项任务 | 重点备赛计划，适合作为桌面主卡片 |

小组件的关键体验改进：

- 显示倒计时、当前阶段、下一项任务、完成数和进度百分比，让计划不用打开 App 也可见。
- 多个进行中计划会自动轮换，避免只展示第一项计划。
- 点击小组件直接进入对应备赛计划详情页，形成“桌面查看 → App 内处理”的回流。
- 支持明暗模式调色和不同尺寸重排，避免在系统桌面上显得像外来控件。
- 无计划时显示空态，避免用户看到空白组件。

设计取舍上，小组件不承担复杂编辑。它只做提醒、概览和回流；真正的任务编辑、AI 调整和归档仍在 App 内完成，减少桌面误触和信息拥挤。

### 2. 系统日历：把关键截止放进用户已有时间系统

计划详情页展示报名截止、提交截止或比赛开始等节点。用户可以一键加入系统日历。

日历集成的体验细节：

- 有日历权限时，App 会写入全天事件，标题包含竞赛名和节点名称。
- 没有权限或写入失败时，App 会打开系统日历 App 的新建事件界面，由用户确认保存。
- 日期通过 ISO 日字符串跨 Flutter 与 Android 传递，原生侧按全天事件处理，降低时区导致日期漂移的风险。
- 旧计划如果缺少报名截止，可以在详情页补填，再加入日历。

这项能力的意义是让“重要节点”进入用户已经习惯查看的系统日历，而不是只存在于 SchoNavi 的计划详情页。

### 3. 通知与截止提醒：把计划推进变成当天动作

Android 端已经接入备赛通知与截止提醒能力。备赛计划会同步成原生可读取的提醒快照，供通知、小组件和闹钟调度使用。

通知侧覆盖三类场景：

- 每日备赛任务提醒：根据未完成任务选择最需要推进的一项。
- 今日摘要：提示今天剩余任务、未来 30 天截止数量和最近截止竞赛。
- 竞赛截止提醒：在截止前 7 天、3 天和当天触发提醒。

任务通知提供三个动作：

- 完成此任务：直接从通知完成任务，并同步刷新快照、小组件和后续提醒。
- 稍后提醒：延后一小时再次提醒。
- 查看计划：点击回到对应备赛计划详情页。

这里的交互重点是减少“看到提醒后还要找入口”的成本。用户可以在通知上完成轻量动作，也可以点击回到计划详情做深度管理。

实际触达仍受系统通知权限、系统通知开关和设备厂商后台策略影响；这些属于 Android 系统层约束，App 需要提供状态提示和降级路径。

## 六、个人档案、收藏、历史与反馈

### 1. 个人档案

用户可以填写学校、专业、阶段、GPA、排名、研究兴趣、竞赛经历和科研经历等信息。档案用于提升导师推荐、匹配分析、套磁邮件和备赛计划的个性化程度。

App 会展示档案完成度，让用户知道哪些信息会影响推荐质量。缺少档案时，相关功能会引导用户补充，而不是直接生成低质量结论。

### 2. 收藏

用户可以收藏感兴趣的导师，后续统一查看和管理。收藏适合用于建立候选导师池，配合导师对比、匹配分析和套磁邮件使用。

### 3. 历史

系统保留用户的搜索与对话历史，方便回到之前的推荐结果和追问上下文。对评委演示时，历史功能也能展示系统支持连续使用，而不是一次性问答。

### 4. 反馈

用户可以对 AI 回复或推荐卡片提交反馈，例如“推荐不准”“信息不准确”“不相关”等。反馈本身不打断主流程，但为后续数据、提示词和推荐策略优化留下依据。

这个设计体现了产品对 AI 不确定性的处理方式：不假设所有输出都正确，而是允许用户在具体结果上给出可追踪反馈。

## 产品亮点

- 完整闭环：从目标输入到推荐候选、行动材料、备赛计划和系统级提醒。
- 可解释推荐：导师和竞赛推荐强调依据，不只输出名单。
- 个性化：结合用户档案、兴趣、经历、时间投入和水平诊断生成建议。
- 可控 AI：计划改动采用“改动卡 + 用户确认”机制。
- 原生触达：Android 小组件、日历和通知把备赛计划延伸到系统层。
- 渐进降级：权限不足或设备不支持时，提供手动添加、日历 App 确认等回退路径。
- 真实学习场景：覆盖找导师、写邮件、选竞赛、做计划、看截止、推进任务等高校学生高频需求。

## 使用边界

- 推荐结果用于辅助筛选，不代表录取、获奖或比赛结果保证。
- 套磁邮件是草稿，用户发送前应自行检查事实、措辞和个人信息。
- 匹配分析是信息性分析，不预测录取概率。
- 备赛计划可帮助组织时间，但仍需要用户根据课程、考试、队友和比赛规则变化持续调整。
- Android 小组件、日历和通知受系统版本、权限、桌面启动器和厂商后台策略影响，不能替代用户自己的重要事项备份。
- 日历事件写入或打开日历 App 后，用户仍应检查日期、标题和备注是否符合个人习惯。
