# 1. 项目概述

## 1.1 项目名称

可暂定为：

> **LightGraphRec：轻量级知识图谱增强推荐系统**

---

## 1.2 项目定位

本项目是一个轻量级智能推荐系统，面向比赛和快速原型场景，采用：

- SQLite 存储结构化业务数据；
- ChromaDB 存储文本向量；
- NetworkX 构建内存知识图谱；
- LLM 负责意图理解、推荐解释生成、自然语言转结构化查询；
- FastAPI 提供统一后端接口；
- 前端可选 Vue / React / Streamlit。

系统不采用 Neo4j、Milvus、LangChain、LangGraph 等重型组件，避免部署复杂、响应慢、资源消耗大等问题。

---

## 1.3 核心设计思想

本项目采用：

> **结构化检索 + 语义检索 + 图谱关系检索 + 轻量级重排 + LLM解释生成**

即：

```text
用户输入
  ↓
意图解析
  ↓
SQLite 精准过滤
  ↓
ChromaDB 语义召回
  ↓
NetworkX 图谱扩展
  ↓
Python 规则重排
  ↓
LLM 生成推荐理由
  ↓
返回推荐结果
```

---

## 1.4 系统解决的问题

传统推荐系统的问题：

| 问题 | 本系统解决方式 |
|---|---|
| 冷启动困难 | 利用物品属性、标签、图谱关系做初始推荐 |
| 推荐不可解释 | 返回图谱路径和自然语言解释 |
| 语义理解弱 | 使用 ChromaDB 做文本语义召回 |
| 部署复杂 | 全部采用本地轻量组件 |
| 比赛演示卡顿 | 控制 LLM 调用次数，核心检索走本地 |
| 需求变化频繁 | 模块解耦，接口稳定，支持敏捷迭代 |

# 4. 核心推荐流程设计

## 4.1 推荐流程总览

```text
1. 用户提交推荐请求
   ↓
2. 系统读取用户画像、历史行为
   ↓
3. LLM 或规则解析用户意图
   ↓
4. SQLite 执行结构化过滤
   ↓
5. ChromaDB 执行语义召回
   ↓
6. NetworkX 执行图谱扩展召回
   ↓
7. 合并候选集
   ↓
8. 规则打分与重排
   ↓
9. LLM 生成推荐理由
   ↓
10. 返回推荐列表、推荐理由、图谱路径
```

---

## 4.2 推荐策略

系统采用混合推荐策略。

### 4.2.1 结构化推荐

适合处理：

- 年份；
- 类型；
- 价格；
- 地区；
- 评分；
- 类别；
- 标签；
- 难度等级；
- 时长；
- 库存；
- 用户显式筛选条件。

示例：

```text
用户说：“推荐 2020 年之后评分高的科幻电影”
```

系统应该提取：

```json
{
  "category": "电影",
  "tags": ["科幻"],
  "min_year": 2020,
  "min_rating": 8.0
}
```

然后查询 SQLite。

---

### 4.2.2 语义推荐

适合处理：

- 治愈系；
- 适合失恋；
- 高燃；
- 氛围感；
- 轻松；
- 压抑；
- 适合周末；
- 类似某个作品但不完全同类型。

这类需求不一定能直接用 SQL 查出，因此用 ChromaDB 语义检索。

---

### 4.2.3 图谱推荐

适合处理：

- 用户喜欢 A，推荐与 A 相关的 B；
- 同作者、同导演、同品牌、同风格；
- 多跳关系；
- “喜欢这个的人也喜欢那个”；
- 推荐理由可解释。

图谱路径示例：

```text
User_1001 
  -- liked --> Item_12 
  -- has_tag --> 治愈
  -- has_tag --> Item_36
```

推荐解释：

```text
推荐《Item_36》，因为你之前喜欢的《Item_12》也具有“治愈”标签。
```

---

## 4.3 候选集融合策略

推荐候选来源：

| 来源 | 说明 |
|---|---|
| SQLite | 精准过滤结果 |
| ChromaDB | 语义相似结果 |
| NetworkX | 图谱关联结果 |
| 热门榜 | 兜底推荐 |
| 用户画像 | 个性化偏好推荐 |

候选融合后，每个物品计算综合分。

---

## 4.4 推荐打分公式

建议 MVP 阶段使用简单可解释公式：

```text
final_score =
    0.35 * semantic_score
  + 0.25 * graph_score
  + 0.20 * profile_score
  + 0.10 * popularity_score
  + 0.10 * freshness_score
```

说明：

| 分数 | 含义 |
|---|---|
| semantic_score | 和用户输入的语义相似度 |
| graph_score | 和用户历史兴趣的图谱距离、关系强度 |
| profile_score | 是否符合用户画像偏好 |
| popularity_score | 物品热度 |
| freshness_score | 新鲜度、多样性、长尾补偿 |

---

## 4.5 图谱打分建议

图谱距离越近，分数越高：

| 图谱关系 | graph_score |
|---|---|
| 用户直接喜欢过同类物品 | 1.0 |
| 2-hop 关联 | 0.8 |
| 3-hop 关联 | 0.6 |
| 同标签 | 0.5 |
| 同作者/导演/品牌 | 0.7 |
| 热门社区内 | 0.4 |

# 6. 向量库设计文档

## 6.2 向量文本构造规则

每个物品用于向量化的文本建议由以下字段拼接：

```text
标题 + 类别 + 标签 + 简介 + 关键属性 + 用户评论摘要
```

示例：

```text
标题：星际穿越
类别：电影
标签：科幻、亲情、宇宙、硬核、催泪
简介：讲述人类穿越虫洞寻找新家园的故事。
```

---

## 6.3 ChromaDB Metadata 设计

每条向量需要保存 metadata：

```json
{
  "item_id": 123,
  "title": "星际穿越",
  "category": "电影",
  "tags": "科幻,亲情,宇宙",
  "rating": 9.3,
  "release_year": 2014
}
```

---

## 6.4 向量更新策略

当以下字段变化时，需要重新生成向量：

- title；
- description；
- tags；
- category；
- metadata_json；
- 用户评论摘要。

判断方式：

```text
对拼接文本计算 hash
如果 hash 与 item_embeddings.text_hash 不一致，则更新向量
```

## 7.3 图谱关系类型

| 关系 | 含义 |
|---|---|
| liked | 用户喜欢物品 |
| disliked | 用户不喜欢物品 |
| viewed | 用户浏览物品 |
| purchased | 用户购买物品 |
| has_tag | 物品拥有标签 |
| belongs_to | 物品属于类别 |
| created_by | 物品由作者/导演/品牌创建 |
| similar_to | 物品相似 |
| same_community | 同一社区 |
| co_interacted | 被相似用户共同交互 |

---

## 7.4 图谱构建来源

| 来源 | 构建方式 |
|---|---|
| 物品标签 | Item -> Tag |
| 物品类别 | Item -> Category |
| 用户行为 | User -> Item |
| 作者/导演/品牌 | Item -> Author/Director/Brand |
| 相似物品 | Item -> Item |
| 用户相似行为 | User -> User 或 Item -> Item |

---

## 7.5 图谱加载策略

服务启动时：

```text
1. 从 graph_edges 表读取所有边
2. 初始化 NetworkX Graph
3. 按 source_id、target_id、relation、weight 加载边
4. 缓存在全局 GraphService 中
```

---

## 7.6 图谱更新策略

当用户产生新行为：

```text
1. 写入 interactions 表
2. 写入 graph_edges 表
3. 同步更新内存中的 NetworkX 图
4. 异步更新用户画像
```

# 16. 各代码文件书写思路说明

> 注意：以下不提供具体代码，只说明每个文件应该承担的职责和实现思路。

---

## 16.1 app/main.py

职责：

- 创建 FastAPI 应用；
- 注册全局路由；
- 注册异常处理器；
- 注册启动事件；
- 启动时加载图谱；
- 启动时初始化 ChromaDB；
- 配置 CORS。

不要在这里写推荐逻辑。

---

## 16.2 app/core/config.py

职责：

- 读取 `.env` 配置；
- 管理数据库路径；
- 管理 ChromaDB 路径；
- 管理 LLM API Key；
- 管理推荐参数默认值。

建议使用 Pydantic Settings。

---

## 16.3 app/core/response.py

职责：

- 定义统一响应格式；
- 提供 success_response；
- 提供 error_response；
- 保证所有 API 返回结构一致。

---

## 16.4 app/core/exceptions.py

职责：

- 定义业务异常；
- 定义参数异常；
- 定义 LLM 调用异常；
- 定义向量检索异常；
- 定义图谱检索异常。

---

## 16.5 app/api/v1/recommend.py

职责：

- 定义推荐相关接口；
- 接收 RecommendRequest；
- 调用 RecommendationService；
- 返回 RecommendResponse。

不做候选召回、不做排序。

---

## 16.6 app/services/recommendation_service.py

职责：

- 推荐流程总编排；
- 调用意图解析；
- 调用检索服务；
- 调用重排服务；
- 调用解释服务；
- 写入推荐日志。

这是推荐系统主入口，但不直接写数据库查询。

---

## 16.7 app/services/retrieval_service.py

职责：

- 聚合多路召回；
- 调用 SQLite 结构化检索；
- 调用 ChromaDB 语义检索；
- 调用 NetworkX 图谱检索；
- 返回候选集。

---

## 16.8 app/services/ranking_service.py

职责：

- 接收候选物品；
- 按 semantic_score、graph_score、profile_score 等计算总分；
- 去重；
- 多样性控制；
- 排序；
- 截断 top_k。

---

## 16.9 app/services/vector_service.py

职责：

- 初始化 ChromaDB；
- 新增物品向量；
- 更新物品向量；
- 删除物品向量；
- 执行 query_texts 检索；
- 将 ChromaDB 结果转换成统一 Candidate 格式。

# 18. 解耦设计原则

## 18.1 推荐流程解耦

RecommendationService 不应该直接依赖具体实现，而应该依赖接口式服务：

```text
VectorService
GraphService
RankingService
LLMService
```

这样后续如果要替换 ChromaDB 为 Qdrant，只需要改 VectorService，不影响 API 层和推荐流程。

---

## 18.2 数据源解耦

物品数据可能来自：

- CSV；
- JSON；
- Excel；
- SQLite；
- 第三方 API。

因此应统一转换为内部 Item 模型。

---

## 18.3 模型解耦

LLM 不应绑定某一家厂商。

建议设计：

```text
LLMService
  ├── DeepSeekProvider
  ├── OpenAIProvider
  ├── QwenProvider
  └── MockProvider
```

比赛期间如果 API 挂了，可以切换 MockProvider 或模板解释。

---

## 18.4 推荐策略解耦

推荐策略应插件化：

```text
StructuredRetriever
VectorRetriever
GraphRetriever
PopularRetriever
ProfileRetriever
```

每个 Retriever 返回统一 Candidate 结构。

---

## 18.5 响应结构稳定

前端最怕后端字段频繁改。

因此推荐接口响应结构要尽早定死：

```text
recommendations
query_understanding
graph_paths
evidence
latency_ms
```

后续新增字段可以加，但不要删除已有字段。

# 20. 性能优化设计

## 20.1 响应时间目标

| 阶段 | 目标耗时 |
|---|---|
| SQLite 查询 | < 20ms |
| ChromaDB 查询 | < 100ms |
| NetworkX 查询 | < 20ms |
| 排序融合 | < 10ms |
| LLM 意图解析 | 300ms - 1000ms |
| LLM 推荐解释 | 500ms - 1500ms |
| 总响应时间 | 1s - 3s |

如果比赛现场网络不稳定，可以关闭 LLM 解释，使用模板解释，响应可控制在 300ms 内。

---

## 20.2 LLM 降级策略

LLM 调用失败时：

```text
1. 使用规则解析意图
2. 使用模板生成推荐理由
3. 返回本地检索结果
4. 在 debug 中标记 llm_fallback=true
```

---

## 20.3 缓存策略

建议缓存：

| 缓存内容 | 方式 |
|---|---|
| 热门推荐 | 内存变量 |
| 用户画像 | 内存字典 |
| 图谱对象 | NetworkX 全局实例 |
| 物品详情 | LRU Cache |
| 推荐结果 | request 级缓存，可选 |

比赛阶段不一定需要 Redis。
